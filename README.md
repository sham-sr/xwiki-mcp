MCP server for [XWiki](https://www.xwiki.org/) REST API. Lets AI agents (Claude Code, Claude Desktop, etc.) **search, read and write** your wiki — with proper Solr full-text search, not the half-broken legacy endpoint most clients default to.

## Why this one

Most XWiki MCP servers (including ones generated by AI coding tools) wire up `/rest/wikis/{wiki}/search`. That endpoint is HQL-based and effectively searches **page names only** — so a query like `"visibility metric"` returns nothing if no page is literally named that, even when dozens of pages discuss the topic. `xwiki-mcp` defaults to `/rest/wikis/query?type=solr`, which is Solr-backed and indexes actual content with proper relevance ranking.

## Tools

### Read / research (v0.4+)

| Tool | Description |
|------|-------------|
| `search` | Solr full-text (auto-fallback to legacy). `wiki` filter, `excerpt` in results, `suggestions` when empty |
| `list_wikis` | Virtual wikis in scope + default wiki |
| `resolve_url` | Browser URL → `{id, wiki, space, page}` |
| `wiki_status` | Solr document counts per wiki (MCP alternative to `npm run check-solr`) |
| `list_spaces` | Top-level spaces across all wikis in scope |
| `list_pages` | Pages in a space (`wiki` param optional) |
| `get_page` | Page content by `id` (preferred) or `space`+`page`; `max_chars` / `content_offset` for long pages |
| `get_page_children` | Direct child pages (`id` supported) |
| `get_attachments` | List attachments (`id` supported) |
| `get_attachment` | Download text attachment content (.md, .txt, .csv, …) |

**MCP prompt:** `wiki_research` — step-by-step research workflow for agents.

**Agent playbook:** [AGENTS.md](AGENTS.md) ([Русский](AGENTS.ru.md)) — MCP config, search strategy, worked `mywiki` example.

**MCP resource:** `xwiki://wikis` — JSON list of wikis in search scope.

### Write (v0.2+)

| Tool | Description |
|------|-------------|
| `create_page` | Create or update a page (upsert via PUT). XML-safe escaping. |
| `delete_page` | Delete a page |
| `add_comment` | Add a comment to a page |

## Configuration

Environment variables:

```
XWIKI_BASE_URL      # Required. Base URL without /rest (e.g. https://wiki.example.com)
XWIKI_AUTH_TYPE     # basic | token | none  (default: basic)
XWIKI_USERNAME      # For basic auth
XWIKI_PASSWORD      # For basic auth
XWIKI_TOKEN         # For token auth (Bearer)
XWIKI_WIKI_NAME     # Default wiki for get_page(space,page) and write ops (optional; see below)
XWIKI_WIKI_NAMES    # Comma-separated wikis for search/list_spaces (optional; auto-discovered if unset)
XWIKI_REST_PATH     # REST path (default: /rest)
XWIKI_PAGE_LIMIT    # Default page size (default: 50)
```

### Multi-wiki setup

**Two different settings — not duplication:**

| Variable | Role |
|----------|------|
| `XWIKI_WIKI_NAMES` | **Scope** — which wikis to search and list spaces in |
| `XWIKI_WIKI_NAME` | **Default** — which wiki to use when the call has no wiki in `id` (`get_page({ space, page })`, `create_page`, etc.) |

If `XWIKI_WIKI_NAMES` is **not set**, the server calls `GET /rest/wikis` at startup and uses **all** virtual wikis on the instance.

If `XWIKI_WIKI_NAME` is **not set**, the default wiki is `xwiki` when present in the discovered list, otherwise the first wiki alphabetically.

**Minimal config** (auto-discover all wikis, default `mywiki` for writes):

```env
XWIKI_BASE_URL=https://wiki.example.com
XWIKI_WIKI_NAME=mywiki
```

**Explicit scope** (search only selected wikis):

```env
XWIKI_WIKI_NAMES=wiki1,wiki2
XWIKI_WIKI_NAME=mywiki
```

After `search`, open pages with `get_page({ id: "<id from result>" })` — the `id` includes the wiki prefix (`mywiki:...`).

`get_page({ space, page })` without `id` always uses the default wiki (`XWIKI_WIKI_NAME` or resolved default).

## Usage with Claude Code / Claude Desktop

Add to `.mcp.json` (Claude Code, project-scoped) or `claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "xwiki": {
      "command": "npx",
      "args": ["xwiki-mcp"],
      "env": {
        "XWIKI_BASE_URL": "https://wiki.example.com",
        "XWIKI_WIKI_NAMES": "wiki1,wiki2",
        "XWIKI_WIKI_NAME": "mywiki",
        "XWIKI_AUTH_TYPE": "basic",
        "XWIKI_USERNAME": "your-username",
        "XWIKI_PASSWORD": "your-password"
      }
    }
  }
}
```

## Search → read workflow

Recommended agent flow (details: [AGENTS.md](AGENTS.md), MCP prompt `wiki_research`):

1. **`search`** — never guess paths. Use **2–4 keywords** from the user question, not the full sentence.
   - `wiki:"mywiki"` — one virtual wiki.
   - `scope:"title"` — find a section by heading.
   - Exact phrase from page text — **double quotes**: `"phrase from wiki"`.
   - Long questions (5+ words) are auto-quoted for Solr; shorter queries use OR between terms.
2. **`get_page({ id })`** — pass `id` unchanged from search (or from `resolve_url`).
3. **`get_page_children({ id })`** — explore section trees.
4. If empty: read `_search.suggestions` and `_search.solr_q`, then **`wiki_status`**, **`list_wikis`**, **`list_spaces`**.
5. Large page: **`get_page({ id, max_chars: 8000, content_offset: 0 })`** — follow `_content.truncated` for next chunk.

### Search `engine` modes

| `engine` | Behaviour |
|----------|-----------|
| omitted | Solr, then legacy fallback if Solr is empty |
| `"solr"` | Solr only (no fallback) — use to verify indexing |
| `"legacy"` | Page name/title search only |

## Search syntax

Solr engine (default) supports:

- bare terms — full-text match across title (boosted) + content
- `"exact phrase"` — phrase match
- `title:foo` / `name:bar` — restrict to a field
- `AND`, `OR`, `NOT` — boolean operators
- per-call `wiki` — limit to one virtual wiki (e.g. `mywiki`)
- per-call `space` — narrow to one section tree

If Solr isn't indexed on your instance, omit `engine` so search auto-falls back to legacy (title/name). Pass `engine: "legacy"` to skip Solr entirely, or `engine: "solr"` for Solr-only (no fallback).

## Solr index health check

**From MCP:** call `wiki_status` (quick counts per wiki) or `wiki_status({ wiki: "mywiki", quick: false })` for exact count.

**From CLI:** Python script (stdlib only):

```bash
# same env vars as xwiki-mcp
export XWIKI_BASE_URL=https://wiki.example.com
export XWIKI_USERNAME=...
export XWIKI_PASSWORD=...

npm run check-solr
# or
python scripts/check_solr_index.py
```

Load credentials from Cursor `mcp.json` instead of env:

```bash
python scripts/check_solr_index.py --mcp-json ~/.cursor/mcp.json --server xwiki
```

Options:

| Flag | Description |
|------|-------------|
| `--probes` | Also smoke-test `title:*` and `text:*` per wiki (slower) |
| `--insecure` | Skip TLS certificate verification |

Example output:

```
wiki              indexed   status    note
----------------- --------- --------- -----------------------------
wiki1             806       ok
wiki2             113       ok
xwiki             340       ok

Summary: 20/20 wikis have Solr documents
Sum of per-wiki indexed docs: 5195
```

`indexed=0` means the wiki has no Solr documents (not indexed or outside Solr scope). Compare counts with **Administration → Search → Solr** in XWiki.
